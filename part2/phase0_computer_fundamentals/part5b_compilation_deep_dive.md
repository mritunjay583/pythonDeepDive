# Deep Dive — How C, Java, and Python Code Actually Becomes Execution

## The One Rule

**Only machine code runs on the CPU. Period.**

No bytecode, no source code, no "interpreted code" ever touches the CPU directly. The CPU only understands its ISA (x86 or ARM instructions). Everything else is just data being processed BY machine code.

---

## Part 1: C — The Simplest Case

### What Happens

```
source.c ──► compiler ──► machine code (.exe) ──► CPU runs it directly
```

### Hands-On: See It Yourself

Create this file:

```c
// add.c
int main() {
    int x = 4 + 5;
    int y = x + 10;
    return y;
}
```

**Step 1: Compile to assembly (human-readable machine instructions)**
```bash
gcc -S -O0 add.c -o add.s
cat add.s
```

You'll see something like:
```asm
main:
    pushq   %rbp
    movq    %rsp, %rbp
    movl    $9, -4(%rbp)      # x = 4+5 (compiler already computed it!)
    movl    -4(%rbp), %eax
    addl    $10, %eax          # y = x + 10
    movl    %eax, -8(%rbp)
    movl    -8(%rbp), %eax    # return y
    popq    %rbp
    ret
```

**Step 2: Compile to object file (actual binary machine code)**
```bash
gcc -c add.c -o add.o
```

**Step 3: See the machine code bytes**
```bash
objdump -d add.o
```

You'll see:
```
0000000000000000 <main>:
   0:   55                      push   %rbp
   1:   48 89 e5                mov    %rsp,%rbp
   4:   c7 45 fc 09 00 00 00    movl   $0x9,-0x4(%rbp)
   b:   8b 45 fc                mov    -0x4(%rbp),%eax
   e:   83 c0 0a                add    $0xa,%eax
  11:   89 45 f8                mov    %eax,-0x8(%rbp)
  14:   8b 45 f8                mov    -0x8(%rbp),%eax
  17:   5d                      pop    %rbp
  18:   c3                      ret
```

**Those hex bytes (55, 48 89 e5, c7 45 fc...) are what the CPU actually executes.** That's it. No interpreter, no runtime. The CPU reads `55` and knows it means `push %rbp`.

**Step 4: Link and run**
```bash
gcc add.c -o add
./add
echo $?    # prints 19 (the return value)
```

### Summary for C

```
┌──────────────┐     ┌──────────────┐     ┌─────────────────┐
│   add.c      │ ──► │  Compiler    │ ──► │  add (binary)   │
│ (source text)│     │  (gcc)       │     │  (machine code) │
└──────────────┘     └──────────────┘     └────────┬────────┘
                                                    │
                                                    ▼
                                           ┌──────────────┐
                                           │     CPU      │
                                           │ runs directly│
                                           └──────────────┘
```

**Key point:** After compilation, the source code is gone. The CPU never sees `int x = 4 + 5`. It only sees `movl $0x9, -0x4(%rbp)`.

## Part 2: Java — Bytecode + JVM + JIT

### What Happens (Two-Phase)

```
Phase 1 (compile time):  App.java ──► javac ──► App.class (bytecode)
Phase 2 (run time):      JVM loads App.class ──► interprets OR JIT-compiles ──► CPU
```

### Hands-On: See It Yourself

Create this file:

```java
// Add.java
public class Add {
    public static int compute() {
        int x = 4 + 5;
        int y = x + 10;
        return y;
    }
    
    public static void main(String[] args) {
        System.out.println(compute());
    }
}
```

**Step 1: Compile to bytecode**
```bash
javac Add.java
# Creates Add.class — this is NOT machine code!
```

**Step 2: See the bytecode**
```bash
javap -c Add.class
```

You'll see:
```
public static int compute();
    Code:
       0: bipush        9       // push constant 9 (4+5 already computed)
       2: istore_0              // store in local variable 0 (x)
       3: iload_0               // load x
       4: bipush        10      // push constant 10
       6: iadd                  // add top two values on stack
       7: istore_1              // store in local variable 1 (y)
       8: iload_1               // load y
       9: ireturn               // return
```

**This is bytecode — NOT machine code.** The CPU cannot execute `bipush`, `iadd`, `ireturn`. These are instructions for the JVM (a virtual machine).

**Step 3: Run it (JVM interprets/JIT-compiles)**
```bash
java Add
# Prints: 19
```

**Step 4: See the JIT in action**
```bash
java -XX:+PrintCompilation Add
```

You'll see lines like:
```
    34    1       3       Add::compute (10 bytes)
```

This means the JIT compiled `compute()` to native machine code.

**Step 5: See the actual machine code generated by JIT**
```bash
java -XX:+UnlockDiagnosticVMOptions -XX:+PrintAssembly Add
```

(Requires hsdis plugin) You'll see actual x86 assembly that the JIT generated — similar to what gcc produces for C!

### What's Running on the CPU at Each Phase

```
PHASE A — JVM starts up:
  The JVM itself is a C++ program compiled to machine code.
  CPU runs JVM's machine code.

PHASE B — Interpreter mode (cold code):
  JVM's machine code reads your bytecode one instruction at a time.
  
  For "iadd" bytecode, JVM executes something like:
    mov eax, [stack_ptr]       ; pop first value
    add eax, [stack_ptr - 8]   ; add second value
    mov [stack_ptr - 8], eax   ; push result
    add instruction_ptr, 1     ; advance to next bytecode
  
  These are REAL machine instructions — but they're the JVM's code,
  not yours. Your bytecode is just DATA being processed.

PHASE C — JIT mode (hot code, after ~10,000 calls):
  JIT compiler translates your bytecode → machine code:
  
  "iadd" becomes literally: add eax, ecx  (1 instruction!)
  
  Now CPU runs YOUR logic as native machine code.
  The interpreter is bypassed. Same speed as C.
```

### The JVM Is a Program Too

This is the key insight most people miss:

```
┌────────────────────────────────────────────────────────┐
│  The JVM (java binary) is itself compiled C++ code      │
│                                                         │
│  It's machine code that READS your .class bytecode     │
│  and either:                                            │
│    A) Interprets it (JVM's code does work for you)     │
│    B) JIT-compiles it (generates new machine code)     │
│                                                         │
│  Either way: only machine code touches the CPU.        │
└────────────────────────────────────────────────────────┘
```

Think of the JVM like a music box:
- The music box (JVM) is a physical machine (machine code)
- The paper roll (bytecode) contains instructions for the music box
- The paper roll doesn't play music itself — the machine reads it and plays

## Part 3: Python (CPython) — Source + Bytecode + Interpreter

### What Happens

```
program.py ──► CPython (at runtime) ──► bytecode ──► interpreter loop executes it
                                                      (never generates machine code
                                                       from YOUR Python code)
```

### Hands-On: See It Yourself

Create this file:

```python
# add.py
def compute():
    x = 4 + 5
    y = x + 10
    return y

print(compute())
```

**Step 1: See the bytecode**
```bash
python3 -c "
import dis
import add
dis.dis(add.compute)
"
```

Or inline:
```python
import dis

def compute():
    x = 4 + 5
    y = x + 10
    return y

dis.dis(compute)
```

Output:
```
  2           0 LOAD_CONST               1 (9)      # 4+5 already folded
              2 STORE_FAST               0 (x)

  3           4 LOAD_FAST                0 (x)
              6 LOAD_CONST               2 (10)
              8 BINARY_ADD
             10 STORE_FAST               1 (y)

  4          12 LOAD_FAST                1 (y)
             14 RETURN_VALUE
```

**This looks similar to Java bytecode!** But the crucial difference: **Python never JIT-compiles this to machine code** (in CPython).

**Step 2: See the .pyc file (cached bytecode)**
```bash
python3 -c "import add"    # creates __pycache__/add.cpython-3x.pyc
python3 -c "
import marshal, dis, sys

with open('__pycache__/add.cpython-311.pyc', 'rb') as f:
    f.read(16)  # skip header
    code = marshal.load(f)
    dis.dis(code)
"
```

**Step 3: See what the CPU ACTUALLY executes**

The CPU runs CPython's interpreter loop. Here's the real C code (simplified from `Python/ceval.c`):

```c
// This is CPython's main loop — COMPILED C CODE (machine code on disk)
PyObject* _PyEval_EvalFrameDefault(PyFrameObject *f) {
    
    for (;;) {
        // FETCH: read next Python bytecode instruction
        opcode = NEXTOP();
        
        switch (opcode) {
            
            case LOAD_CONST:
                value = constants[oparg];
                PUSH(value);       // push onto Python stack
                Py_INCREF(value);  // reference counting!
                break;
            
            case BINARY_ADD:
                right = POP();
                left = TOP();
                
                // Type checking at runtime!
                if (PyLong_Check(left) && PyLong_Check(right)) {
                    result = PyLong_Add(left, right);
                } else if (PyUnicode_Check(left) && PyUnicode_Check(right)) {
                    result = PyUnicode_Concat(left, right);
                } else {
                    result = PyNumber_Add(left, right);  // generic fallback
                }
                
                Py_DECREF(left);   // reference counting!
                Py_DECREF(right);
                SET_TOP(result);
                break;
            
            case STORE_FAST:
                value = POP();
                locals[oparg] = value;
                break;
            
            case RETURN_VALUE:
                retval = POP();
                return retval;
        }
    }
}
```

**THIS is what the CPU runs.** Not your `x = 4 + 5`. The CPU runs this C code's machine-code version, which happens to read your bytecode and do work.

**Step 4: Count the overhead**

For your simple `x + 10` (BINARY_ADD):
```
C (compiled):    1 machine instruction: add eax, 10

Java (JIT'd):   ~1-3 machine instructions (after JIT optimization)

Python:          ~50-100 machine instructions:
                 • Read opcode from bytecode array
                 • Switch/dispatch to BINARY_ADD handler
                 • POP right operand (check stack bounds)
                 • POP left operand
                 • Check type of left (is it int? float? string?)
                 • Check type of right
                 • Call PyLong_Add (which itself does overflow checks, 
                   arbitrary precision math, memory allocation)
                 • Decrement reference count of left
                 • Check if left's refcount is 0 (maybe free it)
                 • Decrement reference count of right
                 • Check if right's refcount is 0
                 • Push result onto stack
                 • Advance bytecode pointer
                 • Jump back to top of loop
```

**That's why Python is 50-100x slower than C for pure computation.**

## Part 4: Visual Comparison

### What's On Disk

```
C:      program.exe     → actual machine code (x86 instructions)
Java:   App.class       → bytecode (JVM instructions, not machine code)
Python: script.py       → source text (not even bytecode yet)
        __pycache__/*.pyc → bytecode (cached, generated on first import)
```

### What Runs on CPU

```
C:
  ┌─────────────────┐
  │ YOUR machine    │ ──► CPU
  │ code directly   │
  └─────────────────┘

Java (cold):
  ┌─────────────────┐          ┌──────────────┐
  │ JVM's machine   │ reads    │ Your bytecode│
  │ code (interp.)  │◄─────────│ (as data)    │
  └────────┬────────┘          └──────────────┘
           │
           ▼
          CPU

Java (hot, after JIT):
  ┌─────────────────┐
  │ JIT-generated   │ ──► CPU  (almost same as C now!)
  │ machine code    │
  └─────────────────┘

Python (always):
  ┌─────────────────┐          ┌──────────────┐
  │ CPython's       │ reads    │ Your bytecode│
  │ machine code    │◄─────────│ (as data)    │
  │ (the interpeter)│          └──────────────┘
  └────────┬────────┘
           │
           ▼
          CPU
           
  Your Python code is NEVER compiled to machine code.
  CPython's pre-compiled C code does all the work.
```

## Part 5: The Critical Difference — Java vs Python

Both have bytecode. Both have an interpreter initially. So why is Java faster?

```
                    Java                          Python (CPython)
                    ────                          ──────
Bytecode:           Yes (.class)                  Yes (.pyc)
Interpreter:        Yes (initially)               Yes (always)
JIT Compiler:       YES! (generates machine code) NO! (never generates machine code)
Type system:        Static (int x = 5)            Dynamic (x = 5, could be anything)
After warmup:       Native speed (~C)             Still interpreted (50x slower than C)
```

**Why can Java JIT but Python can't (easily)?**

```java
// Java: compiler KNOWS x is int, y is int, result is int
// JIT can directly emit: add eax, ecx (1 instruction)
int x = 4;
int y = 5;
int result = x + y;
```

```python
# Python: x could be ANYTHING at runtime
# Interpreter must check types EVERY TIME
x = 4       # int now, but could be reassigned to "hello" next line
y = 5       # same
result = x + y   # Is this int+int? str+str? list+list? Check at runtime!
```

Java's static types let the JIT generate specialized machine code confidently. Python's dynamic types mean the interpreter must always check "what type is this?" before every operation — that check itself is dozens of machine instructions.

**PyPy exists to fix this:** PyPy is an alternative Python runtime WITH a JIT. It observes "x is always int here" and generates specialized machine code. That's why PyPy is 5-50x faster than CPython.

## Part 6: See It Live — Benchmarking

Try this yourself:

```python
# benchmark.py
import time

def add_loop():
    total = 0
    for i in range(10_000_000):
        total += i
    return total

start = time.time()
result = add_loop()
elapsed = time.time() - start
print(f"Python: {elapsed:.3f}s, result={result}")
```

```java
// Benchmark.java
public class Benchmark {
    public static long addLoop() {
        long total = 0;
        for (int i = 0; i < 10_000_000; i++) {
            total += i;
        }
        return total;
    }
    
    public static void main(String[] args) {
        long start = System.nanoTime();
        long result = addLoop();
        double elapsed = (System.nanoTime() - start) / 1e9;
        System.out.printf("Java: %.3fs, result=%d%n", elapsed, result);
    }
}
```

```c
// benchmark.c
#include <stdio.h>
#include <time.h>

long add_loop() {
    long total = 0;
    for (int i = 0; i < 10000000; i++) {
        total += i;
    }
    return total;
}

int main() {
    clock_t start = clock();
    long result = add_loop();
    double elapsed = (double)(clock() - start) / CLOCKS_PER_SEC;
    printf("C: %.3fs, result=%ld\n", elapsed, result);
    return 0;
}
```

**Run them:**
```bash
# C
gcc -O2 benchmark.c -o bench_c && ./bench_c

# Java
javac Benchmark.java && java Benchmark

# Python
python3 benchmark.py
```

**Typical results:**
```
C:      0.008s   (direct machine code, loop optimized away partially)
Java:   0.012s   (JIT compiles the loop to native, nearly same as C)
Python: 0.650s   (interpreter overhead per iteration, 50-80x slower)
```

## Summary Table

| | C | Java | Python (CPython) |
|---|---|---|---|
| **What's on disk** | Machine code | Bytecode (.class) | Source / bytecode (.pyc) |
| **Who runs it** | CPU directly | JVM (interpret + JIT) | CPython interpreter |
| **Machine code from YOUR code?** | Yes (at compile time) | Yes (at runtime, for hot code) | Never |
| **Type info available?** | At compile time | At compile time | Only at runtime |
| **Can optimize?** | Compiler optimizes once | JIT optimizes hot paths (even better than C sometimes) | Minimal optimization |
| **Startup speed** | Instant | Slow (JVM boot + warmup) | Fast (no compilation) |
| **Steady-state speed** | Fastest | Near C (after JIT) | 50-100x slower |
| **Why it's fast/slow** | No overhead | JIT eliminates overhead for hot code | Type checks + dispatch + refcounting per operation |
| **Best for** | Systems, OS, embedded | Servers, long-running apps | Scripting, glue code, prototyping |

## The Mental Model

```
MACHINE CODE (what CPU understands):
  add eax, 10        ← 1 instruction, 1 cycle

C compiles to this ONCE, ahead of time.
Java JIT-compiles to this AT RUNTIME, for hot code.
Python NEVER compiles your code to this. Instead...

PYTHON INTERPRETER (machine code that reads YOUR bytecode):
  // ~50 machine instructions to accomplish the same "add 10":
  read_bytecode()
  dispatch_opcode()
  pop_stack()
  check_type()
  check_type()
  call_PyLong_Add()
    check_overflow()
    allocate_new_int_object()
    compute_result()
  decref_old()
  decref_old()
  push_stack()
  advance_pc()
  goto_loop_top()
```

**The CPU is always running machine code. The only question is: is it YOUR logic compiled to machine code, or is it a GENERIC INTERPRETER's machine code that processes your bytecode as data?**
